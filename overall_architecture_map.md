## 1. EXECUTION MODEL (CRITICAL)
OpenGarrison uses a classic **Authoritative Server with Client-Side Prediction** model. The codebase is heavily separated into client presentation and server authority, with a shared deterministic simulation core.

**Server Execution (`Server/Runtime/GameServer.cs`)**
The server runs a strict, headless update loop (`RunMainLoop`) governed by a high-resolution timer.
1. **Network Pump:** Reads UDP/WebSocket packets and parses client inputs/RPCs.
2. **Simulation Batch (`ServerSimulationBatch`):** Applies accumulated inputs to the `SimulationWorld` and advances the physics/gameplay state by one or more fixed time steps.
3. **Snapshot Broadcast:** Captures the delta-compressed state of the `SimulationWorld` (positions, health, events) and blasts it to all connected clients.

**Client Execution (`Client/Game/Core/Game1.cs`)**
The client runs a variable-framerate rendering loop (Monogame).
1. **Input Capture:** `FrameController` samples hardware mouse/keyboard state.
2. **Network Pump:** Parses incoming server snapshots into an interpolation buffer.
3. **Prediction & Simulation:** The client applies local inputs to its own local copy of `SimulationWorld` (Prediction) while interpolating visual representations between the last known server snapshots (Interpolation).
4. **Rendering:** `GameplayController` and `Game1.GameplayWorldDrawController` render the interpolated state to the screen.

## 2. CORE SYSTEM RESPONSIBILITY MAP

The repository is divided into four main pillars:

### A. Core (`/Core`)
The heart of the game. It contains the deterministic game rules and state.
- **`SimulationWorld`:** The master state container. Holds all entities, the map, score, and the physics/collision runtime.
- **`Entities`:** Definitions for Players, Projectiles, Structures, etc.
- **`Level`:** The collision mesh and navigation graph of the map.
- *Strict Rule:* The Core cannot depend on Monogame (Client rendering) or UDP (Server networking).

### B. Protocol (`/Protocol`)
The translation layer.
- **`ProtocolCodec`:** Handles serialization and deserialization of the game state into bytes.
- **`MessageTypes`:** Defines RPCs and structural packets.
- **`SnapshotDelta`:** The mechanism for delta-compressing the world state so the server only sends what changed.
- **`DemoRecording`:** Replay file format writing and reading.

### C. Server (`/Server`)
The orchestrator of the multiplayer experience.
- **`GameServer`:** The entry point. Handles the socket layer, player authentication, rate limiting, and the tick clock.
- **`ServerPluginRuntime`:** A highly extensible hook system that allows external plugins or server commands (like RCON or chat commands) to mutate the `SimulationWorld` without recompiling the game.
- **`ServerBotManager`:** Responsible for instantiating AI and injecting their synthesized inputs into the input buffer just before the simulation ticks.

### D. Client (`/Client`)
The visual and interactive frontend.
- **`Game1`:** The Monogame application lifecycle.
- **`BrowserInputBridge`:** Special shims for running the game inside a web browser (WASM).
- **`GameplayController`:** Coordinates all visual feedbackâ€”bullet tracers, gore, smoke, UI overlays, and HUDâ€”based on the events generated by the simulation.

## 3. CONTROL FLOW & DATA PIPELINE

1. **Client -> Server (Inputs):**
   Client captures `W, A, S, D, MouseClick`. Packs it into a `PlayerInputSnapshot` and sends it via UDP.
2. **Server Buffer:**
   Server places the input into a queue for that specific player slot.
3. **Server Simulation:**
   Server ticks `SimulationWorld.AdvanceOneTick()`, resolving movement, raycasts, and damage.
4. **Server -> Client (State):**
   Server computes a `SnapshotDelta` (e.g., "Player 1 lost 50 HP", "Player 2 moved to X,Y") and broadcasts it.
5. **Client Interpolation:**
   Client receives the snapshot, rewinds its local prediction, verifies if its prediction was correct, corrects it if not (Rubberbanding), and smoothly interpolates the rendering cameras and sprites to match the server.

## 4. HOTSPOTS (HIGH-RISK AREAS)

- **Snapshot Delta Encoding (`ProtocolCodec.Snapshot.cs`)**
  - *Risk:* If a new field is added to a player in `Core`, but the developer forgets to add it to the Protocol Codec, the client will desync or crash trying to read misaligned bytes.
- **Client Prediction (`Game1.InterpolationTracking.cs`)**
  - *Risk:* The client runs the exact same `SimulationWorld.AdvanceOneTick()` as the server. If the client's local physics math uses floating-point instructions that differ slightly from the server's platform (e.g., WASM vs x64), they will slowly drift out of sync, causing micro-stutters.
- **Server Tick Starvation (`ServerSimulationBatch.cs`)**
  - *Risk:* If `AdvanceOneTick()` takes longer than the real-time tick window (e.g., >33ms for a 30TPS server), the server will drop the backlog and players will experience severe lag or skipped inputs.

## 5. MINIMAL MENTAL MODEL

To understand OpenGarrison:
1. Everything physical and logical happens in `Core` (`SimulationWorld`).
2. `Server` is just a network pump that owns the "real" `SimulationWorld` and pushes updates down via `Protocol`.
3. `Client` is a glorified rendering engine that owns a "fake" `SimulationWorld`, guessing what will happen based on local inputs, and correcting itself when the server tells it otherwise.
