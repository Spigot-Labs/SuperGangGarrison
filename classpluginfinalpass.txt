What The Three Need
RebelINS is the cleanest first target. INS defines it as Nene/CLASS_REBEL: 120 HP, speed 1.4, double jump, wall jump/slide, RebelShotty, 6-shell scattergun, a regenerating 0-100 charge meter, and an aim-direction dash costing 40 charge. Current weapons/abilities can cover the scattergun, dash, cooldown, and HUD meter. Missing for fidelity: wall slide/wall jump and possibly team handling, since INS only maps RebelINS to red-side NeneRed.
Ash is a good second target. INS defines Ash as 160 HP, speed 1.05, Boomstick, 2 shells, held chainsaw using gas, chainsaw stance sprites, and EvilAsh summon on a long cooldown. Current seams cover the shotgun, held special, cooldown state, HUD, damage, and impulses. Missing for fidelity: summon/minion support and class animation override for chainsaw stance.
Dante is the stress test. INS defines 130 HP, speed 1.09, double jump, damage resistance, DanteArsenal, 3 firearms, melee combo chains, roll/air dash, charge attacks, Devil Trigger resource, self-heal, custom hitboxes, and a class-specific HUD. Current seams can approximate weapons, resource state, HUD, self-heal, status effects, and stock projectile spawning. Missing for fidelity: authoritative arbitrary ability buttons, custom melee hitboxes, richer projectile definitions, and ability-driven animation states.
Current Gaps
The biggest class-select gap is that plugin classes are listed/mouse-selectable, but plugin hotkeys are not data-driven. Stock keys and Quote are hardcoded in Game1.ClassSelect.cs (line 40), while plugin entries are just enumerated later in the same file. We’d need class metadata like classSelectHotkey: "A" and client handling for A/D/R.
The enablement gap is more serious. If a gameplay class is registered, the server accepts its gameplay class ID in SelectClass today in ServerSessionManager.cs (line 372). There is no “registered but disabled/hidden” state. To ship these disabled by default, we need a server-authoritative class availability gate, mirrored by the client UI.
Console support is enum-only right now. Server class only parses PlayerClass in ServerBuiltInCommandRegistrar.cs (line 248), and practice set_class is also enum-only in Game1.Console.cs (line 583). We’d need commands such as set_gameplay_class <id|alias> and enable_gameplay_class <id> <on|off>.
Ability/HUD foundations are decent. Server Lua already exposes replicated state, damage/heal/impulse/status, stock projectile spawning, custom ability executors, and custom primary weapon behavior in LuaServerPlugin.cs (line 1205). Client Lua can draw HUD widgets and read replicated state, but it mostly exposes legacy PlayerClass, not GameplayClassId, in LuaClientPlugin.cs (line 221).
Input is the limiting generality issue. Network input has primary, secondary, utility, interact, swap, taunt, and movement buttons in PlayerInputSnapshot.cs (line 3) and MessageTypes.cs (line 54). That can approximate these classes, but Dante-style arbitrary hotkeys/actions need a generalized authoritative plugin ability input channel.
Recommendation
Use RebelINS first to prove the class availability gate, A/D/R hotkey metadata, replicated HUD meters, and a custom primary/utility ability. Then Ash to force held melee plus animation override. Do Dante last, because it will expose the need for generalized action inputs, custom melee hitboxes, richer projectile definitions, and ability-driven presentation.
