Executable edge state machine
Each certified edge should run as:
Stage -> Commit -> Launch/Drop/Fall -> Complete -> Fail
with explicit failure reasons. Current steering still blends staging, jumping, recovery, and waypoint pursuit together.

Pathfinder awareness of live-executable edges
The graph says “this edge was probe-certified,” but not “this bot, from this live approach state, can execute it right now.” If a live edge fails its recipe, the router should temporarily blacklist that edge/context and route around it.

Staging behavior
When the bot enters an edge wrong, it needs to stage into the launch recipe: get grounded, get into the launch X/Y window, face/move the expected direction, reach speed, then commit. Right now it still often tries to blend target seeking with launch prep.

Macro traversal segments
The stair return is not one jump. It is a small corridor sequence. Treating it as isolated edges means each edge can be “correct” but the chain still fails. We likely need certified multi-edge movement segments for stair/platform corridors.

Surface-aware recovery
When a bot lands on the wrong adjacent surface, it should know whether it can continue, retry, or must repath. Current recovery mostly reacts to position and waypoint distance, not semantic surface state.
